# Implementation Plan: Simple Product Catalog Website

**Branch**: `002-product-catalog-mvp` | **Date**: 2026-04-29 | **Spec**: specs/002-product-catalog-mvp/spec.md
**Input**: Feature specification from `/specs/002-product-catalog-mvp/spec.md`

## Summary

Deliver a fast, reliable MVP product catalog website with exactly two public
pages (Home and Catalog), in-page product detail, category filtering, friendly
error states, and responsive styling. Keep implementation intentionally small,
with static seed data and no admin UI, while enforcing engineering workflow
requirements: mutation testing, automated repository setup checks, GitHub
Actions deployment, and basic monitoring verification.

## Technical Context

**Language/Version**: TypeScript 5.x (strict mode), Node.js 20 LTS  
**Primary Dependencies**: Next.js 14, React 18, ESLint, Playwright, Vitest  
**Storage**: Static seed data file (JSON/TS module), no external database  
**Testing**: Vitest (unit), Playwright (smoke/e2e), StrykerJS (mutation)  
**Target Platform**: Web browsers (mobile + desktop), Vercel or static host via GitHub Actions  
**Project Type**: Single web application  
**Performance Goals**: First catalog render under 2s locally; no blocking runtime fetches for seed data  
**Constraints**: Two pages only, no cart/checkout/accounts/admin UI, minimal complexity  
**Scale/Scope**: 10 sample products, 3-5 categories, one feature branch delivery

## Data Model

### Product

- `id: string` (stable unique id, e.g., `p-001`)
- `name: string` (1-80 chars)
- `category: Category`
- `shortDescription: string` (1-120 chars)
- `fullDescription: string` (1-500 chars)
- `imageUrl: string` (absolute or root-relative path)

Validation rules:
- Required fields: all
- `category` must match allowed enum values
- Missing/invalid `imageUrl` falls back to placeholder image

### Category

Enum values (small fixed set):
- `Office`
- `Home`
- `Outdoor`
- `Accessories`

### Catalog Query State

- `activeCategory: Category | "All"`

## Seed Dataset Plan

- Use exactly 10 products in `src/data/products.ts`.
- Ensure each category has at least 2 products except one category with 1 item
  to validate edge filtering behavior.
- Include one product with intentionally broken image URL in non-production seed
  test mode to validate fallback handling.

## Public Pages and Navigation

- Home (`/`): business headline, short value proposition, CTA to Catalog.
- Catalog (`/catalog`):
  - category filter control
  - responsive product grid
  - in-page detail panel for selected product
  - friendly empty/no-results states
- Navigation: simple top nav with `Home` and `Catalog` only.

## Optional Product Management Path

- Default decision: exclude admin/product management UI for MVP.
- Lightweight fallback path if urgently requested: edit seed data file
  `src/data/products.ts` and redeploy through CI.
- No runtime admin route, no role model, no auth.

## Validation and Error Handling

Validation:
- Validate seed data schema at startup/build via `validation.ts`.
- Ignore invalid product records only in local debug mode; fail build in CI.

Friendly error states:
- Empty dataset: show "No products available right now" with retry/home link.
- No category matches: show "No products in this category" with reset option.
- Broken image: show placeholder image and alt text.
- Unexpected render/data error: show friendly generic error panel and retry CTA.

## Responsive Styling Strategy

- Mobile-first CSS with breakpoints around 768px and 1280px.
- Product grid:
  - 1 column mobile
  - 2 columns tablet
  - 3-4 columns desktop
- Detail panel:
  - stacked below list on mobile
  - side panel on desktop
- Accessibility baseline:
  - visible focus styles
  - semantic headings and landmarks
  - keyboard-accessible filter and product selection

## Test Approach

Unit tests (Vitest):
- category filtering behavior
- product validation rules
- image fallback helper behavior

E2E smoke tests (Playwright):
- Home -> Catalog navigation works
- category filter narrows products
- selecting product opens detail panel
- no-results and reset behavior visible

Mutation testing (StrykerJS):
- Run against filtering and validation units.
- Gate target: mutation score >= 70% for covered modules.
- If below threshold, improve tests before release candidate.
